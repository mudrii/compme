## A0 Exit Decision (spec §9)

Status of probes: P1 PASS, P2 PASS (34-36ms warm 12-tok, suggest() ~38ms warm), P3 PASS (AX text read in TextEdit, Safari, Chrome), P4 PASS (usable caret rects in TextEdit, Safari, Chrome; movement tracked), P5 PASS (active tap swallows Tab and passes other keys), P5b PASS (listen-only observer + consuming tap split proven), P6 PASS (visible layer-101 non-activating click-through overlay verified), P7 PASS (read→infer→overlay-at-caret metadata verified).

- Caret geometry works in ≥1 real app: PASS — P4 verified TextEdit/Safari/Chrome, movement tracked in TextEdit.
- Two-tap CGEventTap proven stall-free: PASS — P5 active consumer swallowed Tab and passed other keys; P5b proved observer/consumer split with false-state pass and true-state swallow.
- NSPanel overlay shows non-activating + click-through: PASS — P6 screenshot showed visible ghost text, CoreGraphics proved an onscreen layer-101 panel, frontmost-app checks showed no focus steal, and Chrome click-through changed `clicked-0` to `clicked-1` through the overlay.
- Warm inference latency: instruct integration warm **34-36ms** (12 tok), model-compare raw/terse completions **47-52ms** total across instruct/base on `llama-cpp-2-metal` — comfortably under the 500ms floor (P2, automated). ✓
- Base vs Instruct for autocomplete: **resolved for A1a development** — use `models/qwen2.5-0.5b-q4_k_m.gguf` with the terse continuation prompt and candidate shaping. Empty-suffix FIM is rejected in this spike; instruct remains latency-proven fallback, but is not the default because its raw autocomplete quality drifted more often.
- llama.cpp+Metal build complexity: ~43s first build, no extra flags, 25/25 layers to GPU (P1). ✓

**GO / NO-GO for A1:** GO for contract-first A1a implementation and A1b planning against `docs/superpowers/plans/2026-06-04-a1b-macos-adapter-contract.md`. Core risks are now reduced enough to start implementation: AX text/context, caret geometry, event taps, split observer/consumer taps, visible click-through NSPanel overlay, Metal inference, read→infer→overlay metadata, and the A1a model default have all been exercised. Guardrail: A1a must implement the contract-first revisions, not the old narrow platform trait.

**Carry into A1 (key A0 findings so far):**
- **macOS-FFI signature drift vs the draft probe code (all fixed to compile against the pinned versions; do NOT bump versions):**
  - accessibility-sys 0.2.0: `AXIsProcessTrusted` and `AXValueGetValue` return `bool` (use `!trusted` / the bool directly, not `== 0` / `!= 0`). All other AX symbols/sigs matched the draft. `kAXValueType*` are `u32` (`AXValueType`); AX attribute-name consts are `&str`; `CFRange`/`CFTypeRef` are re-exported from `core_foundation::base`.
  - core-graphics 0.25.0 `CGEventTap`: callback returns `CallbackResult` (`Keep`/`Drop`/`Replace`), NOT `Option<CGEvent>`; `mach_port` is private → `tap.mach_port()`; `CFRunLoop::add_source` is a SAFE fn now; keycode field key is `EventField::KEYBOARD_EVENT_KEYCODE` (a const on the `EventField` struct), NOT on `CGEventField` (which is `type = u32`).
  - core-foundation 0.10.1: `kCFRunLoopCommonModes` is re-exported via `core_foundation::runloop` (from `-sys`) but is an extern `static`, so reading it requires `unsafe`.
  - objc2 0.6.4 / objc2-app-kit 0.3.2: `T::alloc(mtm)` requires `objc2::MainThreadOnly` in scope. **NSPanel** init (`initWithContentRect_styleMask_backing_defer`) + the window/control setters are **safe** fns (NSWindow's init is `unsafe`, but NSPanel overrides it safe) → no `unsafe` blocks for the overlay. AppKit types are feature-gated but all needed ones (incl. `NSGraphics`, which gates the panel init) are in the crate's default features, so no `Cargo.toml` feature changes were required. Confirmed-good draft calls: `setLevel(101: NSWindowLevel=isize)`, `NSColor::{clearColor, colorWithWhite_alpha}`, `NSTextField::labelWithString`, `NSScreen::mainScreen(mtm)`, `NSApplicationActivationPolicy::Accessory`, `NSBackingStoreType::Buffered`, `NSWindowStyleMask::{Borderless,NonactivatingPanel}` (bitflags).
- **Coordinate conversion:** AX returns top-left-origin `CGRect`; Cocoa panel placement needs the bottom-left flip via `screen_h - y - h` (lib `ax_to_cocoa_origin`); `NSScreen::frame()` gives `screen_h` (objc2_core_foundation `CGRect`, CGFloat=f64). Retina/multi-monitor pixel-vs-point offset still to be quantified in the manual run.
- **Seam shape validated end-to-end:** `BoundsSource`/`FieldSource`/`Completer` (the A1 `PlatformAdapter` prototype) compile against real AX + llama with the bins as pure glue — the tested lib owns ladder/context/pipeline/coord logic. Spike code now uses `token_to_piece` with a UTF-8 decoder, so A1 should keep that path and not reintroduce deprecated decode APIs.

## Task 5c — Input Monitoring decision (design spec §15 F1 / D1), 2026-06-09

**Probe:** `p8_carbon_hotkey` (above). **Result:** bare Tab (48) and grave (50) both
`RegisterEventHotKey` with **status 0 (noErr)** and no modifiers, with a working
`kEventHotKeyPressed` handler. Carbon hotkeys require only the accessibility-class
entitlement; there is no `CGEventTapCreate`, so **no Input Monitoring TCC prompt**.

**Decision: GO for the Carbon branch** (adopt Carbon `RegisterEventHotKey` for
accept-key interception; drop the consuming `CGEventTap` and the Input-Monitoring
requirement — Accessibility-only, Cotypist parity). Rationale:
1. The one flagged uncertainty — "can bare Tab even be registered as a global
   hotkey?" — is answered **yes** (status 0, this machine/OS).
2. The b73 decompile already proves Cotypist swallows Tab via Carbon hotkeys with
   Accessibility-only and ships **no** CGEventTap (D1/F1), so cross-app consume is
   established by precedent for the exact mechanism.

**Historical residual (superseded by A1b):** at the time of this spike, live
cross-app *consume* + Input-Monitoring-revoked confirmation in our own binary was
still pending. That production migration has since landed: the shipping accept
path uses transient Carbon hotkeys for Tab/grave/Esc/Down and does not require
Input Monitoring for accept behavior. Keep the old two-tap `CGEventTap` probes as
prior-art evidence only; the current source of truth is `docs/ACCEPTANCE.md` and
the macOS MVP design spec's F1/D1 closure notes.
